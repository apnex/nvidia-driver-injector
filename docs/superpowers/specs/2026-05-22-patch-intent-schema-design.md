# Patch-intent schema — design

**Status:** approved design — 2026-05-22. Defines the engineered schema for
per-patch intent specifications, plus the canonical template, the validation
tool, and the index renderer. The actual *content* of the 11 intents and the
subsequent deep code-review are separate cycles that consume this design.

## Context

The injector now ships a complete `C+E+A` composed driver (vanilla
`595.71.05` + `C1`–`C5` + `E1` + `A1`–`A5`). The 11 fork branches on
`apnex/open-gpu-kernel-modules` are load-bearing for project reliability and
are the public artefact the community (issue #979 watchers, future
contributors, eventual NVIDIA reviewers) reads. The `C/E` set is upstream-bound;
the upstream PRs are gated on the soak per `upstream-plan.md`.

For both the upcoming deep-review cycle and the eventual upstream PRs we need
a **canonical, normative, per-patch capability specification** — the patch's
intent as an engineered artefact, not prose conventions. This design defines
that artefact and the tooling that keeps it consistent.

The schema is informed by [OpenSpec](https://openspec.dev)'s spec-driven
approach — specifically its *capability-spec* layer (Purpose, named
Requirements with RFC 2119 keywords, GIVEN/WHEN/THEN scenarios). OpenSpec
deliberately leaves out frontmatter, validation, templates, and tooling; this
design adds them. We borrow the shape; we engineer the rigour.

## Scope

**In scope:**

- The patch-intent schema — YAML frontmatter + ordered Markdown sections.
- A canonical template `_template.md` (lint-conformant skeleton).
- `tools/intent-lint.sh` — the validator.
- `tools/render-patch-index.sh` — the index generator.
- `tools/lib/intent.sh` — shared parsing helpers (mirrors `lib/manifest.sh`).
- `tests/test-intent-lint.sh` — fixture-based tests of every lint rule.
- Directory layout and file-naming conventions.

**Out of scope:**

- The content of the 11 intents — populated in the subsequent deep-review
  cycle that consumes this design.
- Wider "dossier" artefacts — per-patch hunk-by-hunk review notes,
  review-verdict files. These are review-process outputs, designed separately
  if needed.
- The deep-review process itself — subagent shape, gating, tiering of
  reviewer effort. This schema is one *input* to that process.

Filename stem equals manifest id. One intent file per patch — 11 files, plus
the template. Filename ↔ frontmatter id ↔ `patches/manifest` row are
1:1:1 and enforced by `intent-lint`.

## The schema

### YAML frontmatter (7 required fields)

Every intent file begins with frontmatter delimited by `---`:

```yaml
---
id: C3-gpu-lost-retry              # must equal filename stem
layer: base                         # base | addon — must match patches/manifest
source-branch: c3-gpu-lost-retry    # must equal manifest row's fork:<branch>
upstream-candidacy: high            # high | medium | low | n/a
telemetry-tier: mandatory           # mandatory | nominal | none
status: draft                       # draft | reviewed | approved
related-patches: [C5-crash-safety]  # YAML list; entries must resolve
---
```

Field semantics:

| Field | Type / values | Meaning |
|---|---|---|
| `id` | string | Logical patch identity. **Must equal filename stem** and the patch's `patches/manifest` row id. |
| `layer` | `base` \| `addon` | Must match the manifest row's layer column for this id. |
| `source-branch` | string | The fork branch name (without the `fork:` prefix). Must match the manifest's `source: fork:<branch>` field for this id. |
| `upstream-candidacy` | `high` \| `medium` \| `low` \| `n/a` | Likelihood the patch is accepted upstream. `n/a` is the only allowed value for `layer: addon` (addon never upstreams). |
| `telemetry-tier` | `mandatory` \| `nominal` \| `none` | `mandatory` — silent behaviour that's invisible without logs (e.g. `C3` recovery, `A3` recovery). `nominal` — standard prove-the-path-ran logging at kernel-appropriate levels. `none` — no telemetry expected (build metadata, pure libraries). |
| `status` | `draft` \| `reviewed` \| `approved` | Workflow state. |
| `related-patches` | YAML list of ids | Cross-references to other intent files. Empty list (`[]`) allowed. Every listed id MUST resolve to another intent file. |

All 7 fields are required. `intent-lint` rejects missing or malformed fields.

### Required Markdown sections in fixed order

After the frontmatter, the body MUST contain these sections in this order:

1. **`# <id> — <human title>`** — top-level heading. The id portion MUST equal the frontmatter `id`.
2. **`## Purpose`** — one paragraph stating the persistent capability the patch grants the driver. Plain prose; lowercase RFC 2119 keywords if used here are treated as prose, not normative.
3. **`## Requirements`** — one or more `### Requirement: <name>` blocks. Each block contains:
   - a normative paragraph (RFC 2119 keywords UPPERCASE),
   - at least one `#### Scenario: <name>` block written in GIVEN / WHEN / THEN / AND style (Gherkin-flavoured BDD).
4. **`## Scope boundary`** — bullet list of *deliberate* non-goals: things this patch could conceivably touch but does not.
5. **`## Telemetry contract`** — table or list naming each emitted log event, its kernel log level, and the message format (template string).
6. **`## Provenance`** — bullet list with source cluster (which legacy `patches/legacy/000N-*` file is the origin), vanilla baseline file(s) and symbol(s) it modifies, the fork branch name, and any upstream-issue URLs.

`intent-lint` checks presence and order; section bodies are author judgment beyond the structural rules below.

### RFC 2119 conformance (strict)

Normative keywords are the RFC 2119 set: **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**, **SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, **OPTIONAL**.

- Inside `## Requirements` blocks, RFC 2119 keywords MUST appear in UPPERCASE when used normatively.
- Every `### Requirement:` block MUST contain at least one UPPERCASE RFC 2119 keyword.
- Outside `## Requirements`, RFC 2119 keywords appearing in prose use lowercase (they are descriptive, not normative).

### Cross-reference syntax

| Reference target | Form |
|---|---|
| Another patch's intent | `[[<id>]]` — e.g. `[[C5-crash-safety]]`. `intent-lint` resolves each to an intent file. |
| Vanilla NVIDIA source | Backticked `path:symbol` — e.g. `` `kernel-open/nvidia/os-mlock.c:osHandleGpuLost` ``. Not lint-validated (out of scope). |
| Upstream issue | Full URL — e.g. `https://github.com/NVIDIA/open-gpu-kernel-modules/issues/979`. |

## Concrete example

A complete, lint-conformant `docs/patch-intents/C3-gpu-lost-retry.md`:

```markdown
---
id: C3-gpu-lost-retry
layer: base
source-branch: c3-gpu-lost-retry
upstream-candidacy: high
telemetry-tier: mandatory
status: draft
related-patches: [C5-crash-safety]
---

# C3-gpu-lost-retry — GPU-Lost Retry

## Purpose

A transient PCIe dead-bus read on `NV_PMC_BOOT_0` (a single completion-timeout
manifesting as `0xFFFFFFFF`) must not commit the GPU to a permanent
`PDB_PROP_GPU_IS_LOST` state. Headline fix for
[NVIDIA/open-gpu-kernel-modules#979](https://github.com/NVIDIA/open-gpu-kernel-modules/issues/979).

## Requirements

### Requirement: Bounded retry of the dead-bus signature

Before invoking `osHandleGpuLost`, the driver SHALL retry the `NV_PMC_BOOT_0`
read up to `TB_EGPU_GPU_LOST_RETRIES` times at `TB_EGPU_GPU_LOST_DELAY_US`-µs
intervals. If any retry returns a value other than `0xFFFFFFFF`, the driver
MUST treat the original read as transient and continue normal operation.

#### Scenario: Transient bus blip recovers
- **GIVEN** the GPU is healthy and on the PCIe bus
- **WHEN** one read of `NV_PMC_BOOT_0` returns `0xFFFFFFFF`
- **AND** a subsequent read within the retry budget returns a valid value
- **THEN** the driver MUST continue normal operation
- **AND** `PDB_PROP_GPU_IS_LOST` MUST NOT be set

#### Scenario: Genuine GPU loss after exhausted retries
- **GIVEN** the GPU has fallen off the PCIe bus
- **WHEN** all `TB_EGPU_GPU_LOST_RETRIES` retries return `0xFFFFFFFF`
- **THEN** the driver MUST call `osHandleGpuLost`
- **AND** `PDB_PROP_GPU_IS_LOST` MUST be committed

### Requirement: Mandatory recovery telemetry

On every successful retry-recovery the driver SHALL emit one `dev_warn`
log line `"transient bus read recovered after %d retries"`. A
silently-recovered transient is otherwise invisible (this is the C3
mandatory-telemetry rationale).

#### Scenario: A recovery is observable
- **GIVEN** the driver retries the dead-bus signature and recovers
- **WHEN** the recovery completes
- **THEN** exactly one `dev_warn` log line MUST be emitted for the event
- **AND** the line MUST include the recovered-retry count

## Scope boundary

- Covers ONLY the `osHandleGpuLost` preflight path. Dead-bus reads from
  other call sites (`osDevReadReg*`, RPC paths, cleanup paths) are covered
  by [[C5-crash-safety]].
- The retry count and delay are project-tunable constants; their default
  values (10 retries / 100 µs) are review-negotiable per the C3 PR.
- This patch does NOT introduce any module parameter.

## Telemetry contract

| Event | Level | Format |
|---|---|---|
| Retry recovered | `dev_warn` | `"transient bus read recovered after %d retries"` |

No per-retry logging (would be high-frequency on degraded links). One log
line per recovery event.

## Provenance

- **Source cluster:** P1 (`patches/legacy/0001-tb-egpu-gpu-lost-crash-safety.patch`), retry sub-theme.
- **Vanilla baseline:** `kernel-open/nvidia/os-mlock.c:osHandleGpuLost`.
- **Fork branch:** `c3-gpu-lost-retry` on `apnex/open-gpu-kernel-modules`.
- **Upstream issue:** https://github.com/NVIDIA/open-gpu-kernel-modules/issues/979.
```

That is the shape every intent file takes — about 50–80 lines on average.
For 11 patches, expect 500–900 lines of total normative documentation.

## Tooling

### `tools/intent-lint.sh` — validation rules

`intent-lint` validates every `docs/patch-intents/*.md` except `_template.md`.
Rule set:

1. **Frontmatter present, parseable, with all 7 required fields.** Each field has a valid value per the table above.
2. **`id` field equals the filename stem.**
3. **`layer` matches the corresponding `patches/manifest` row's layer.** (Reads `patches/manifest` as ground truth.)
4. **`source-branch` equals the manifest row's `fork:<branch>` value** for the same id, with the `fork:` prefix stripped.
5. **`upstream-candidacy: n/a` ↔ `layer: addon`.** A `base` row must have a non-`n/a` candidacy; an `addon` row must have `n/a`.
6. **Every `related-patches` entry resolves to another `docs/patch-intents/<id>.md` file.**
7. **The required sections appear in the exact order specified** in the schema (`Purpose`, `Requirements`, `Scope boundary`, `Telemetry contract`, `Provenance`). Missing or out-of-order sections fail loud.
8. **`## Requirements` contains ≥ 1 `### Requirement:` heading.**
9. **Each `### Requirement:` block contains ≥ 1 UPPERCASE RFC 2119 keyword.** Detection is via grep of the canonical keyword list.
10. **Each `### Requirement:` block has ≥ 1 `#### Scenario:` block.**
11. **The top-level `# <id> — …` heading's id prefix equals the frontmatter `id`.**

Implementation in plain Bash, sourcing `tools/lib/intent.sh` for shared parsing (frontmatter extraction, section iteration). Mirrors the shape of `tools/lib/manifest.sh` exactly. Fails loud — first failure on each file names the file path, the rule, and the offending content.

### `tools/render-patch-index.sh` — index builder

Reads every `docs/patch-intents/*.md` (excluding `_template.md`). For each, extracts the frontmatter and the first paragraph of `## Purpose`. Emits `docs/patch-index.md`:

```markdown
# Patch index

| Id | Layer | Upstream candidacy | Telemetry | Status | One-line summary |
|---|---|---|---|---|---|
| [C1-kbuild-version-mk](patch-intents/C1-kbuild-version-mk.md) | base | high | none | draft | <first sentence of Purpose> |
| [C2-aer-internal-unmask](patch-intents/C2-aer-internal-unmask.md) | base | medium-high | nominal | draft | … |
| … |
```

Order: manifest order (C1, C2, C3, C4, E1, C5, A1, A2, A3, A4, A5).
Run manually after intent updates; output is committed. (Wiring it into a CI hook is a follow-on if/when CI exists.)

### Tests — `tests/test-intent-lint.sh`

Fixture-based, mirroring `tests/test-manifest-lib.sh`:

- **One valid-fixture case** — a complete, conformant intent file lints clean (exit 0).
- **Eleven violation cases — one per rule above.** Each fixture deliberately breaks one rule and asserts the linter exits non-zero with an error message identifying that rule.

Same harness as the rest of the project (`tests/lib.sh` `assert_eq` / `assert_exit`). The `manifest_lint` test pattern is the model.

## Tiering — C/E vs A

Same schema; different rigour in the **prose** (not the structure):

- **C/E patches (upstream-bound).** Formal phrasing (*"the driver"* rather than *"we"*), every Requirement has Scenarios, Purpose explicitly cites the public issue/RFC where relevant. PR-grade.
- **A patches (project-local).** Scenarios required for every load-bearing Requirement. Project terminology (`TB_EGPU_*`, project-internal constants) acceptable in the normative prose. Still strict RFC 2119 inside Requirements. Operational-grade.

`intent-lint` is **structure-only and layer-agnostic** — it does not enforce a tone register. Tier discipline is reviewer judgment.

## Deliverable — definition of complete

After implementation (separate plan + execute cycle):

- `docs/patch-intents/_template.md` exists, is lint-clean, and demonstrates every required field/section.
- `tools/intent-lint.sh`, `tools/render-patch-index.sh`, `tools/lib/intent.sh` are present and executable.
- `tests/test-intent-lint.sh` passes — covers all 11 lint rules with fixtures.
- `docs/patch-index.md` is generated from the (initially-empty) intents directory or the template alone, proving the renderer works end-to-end before any real intent is populated.
- `tests/run.sh` includes `test-intent-lint.sh` and stays 100% green.

## Out of scope / follow-on

- **Populating the 11 intent files** — its own brainstorm → spec → plan → execute cycle, the deep-review pass. Will consume this schema, run per-patch Opus reviews, and produce the 11 conformant intent files.
- **Wider dossier artefacts** — per-patch review-verdict notes, baseline-anchored hunk-by-hunk reviews if those become useful. Designed if/when the deep-review cycle surfaces a need.
- **`upstream-plan.md` reconciliation** — the one-liner `**Change:**` replacement happens as part of the populate-the-intents cycle (the intent docs must exist before they can be linked).
- **CI integration** of `intent-lint` and the index renderer — when the project gains a CI surface.
- **Cross-reference resolution for vanilla source paths** — checking that `` `path:symbol` `` references resolve to real lines in the base. Deliberately deferred (YAGNI for 11 patches; reviewer can verify by hand).

already-merged dynamic-patch-composition mechanism
([dpc-design](2026-05-22-dynamic-patch-composition-design.md)) and

runs in parallel with `production-migration.md` steps 5–7 (build → soak →
