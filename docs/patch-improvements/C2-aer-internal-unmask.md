## v1 archaeology

What the aorus-5090 mining surfaced about C2's two-stage ancestor chain (G3-H → P5 → C2):

- **Original design intent — empirical discovery via Windows parity (2026-05-07).** `docs/reliability-hypothesis-ledger.md:360` and `patches/0022-G3-H-clear-AER-UncMask-match-Windows.patch:7-15` jointly document the genesis: "Windows `DEVPKEY_PciDevice_Uncorrectable_Error_Mask = 0`; Linux observed `0x00400000`" was the trigger observation. The PCIe r6.0 §6.2.3.2 demotion-to-Advisory-Non-Fatal mechanism (status not latched, Header Log not captured, `pci_error_handlers` not invoked) was identified from the spec **independently** as the operational consequence. The cross-OS comparison was *how the regression was found*; the spec-based mechanism is *why the fix is correct*. v2 intent's Purpose section names the spec mechanism, not the Windows comparison — which is the right framing for an `upstream-candidacy: high` patch (Windows parity is a project-internal anchor, not an upstream argument).
- **Constraint discovered — the "stale RW1C" finding inverts the obvious-but-wrong narrative.** `docs/lever-catalog.md:863-869` and `docs/h17-g3-gen3-investigation-2026-05-07.md:18-23` (the executive-summary reframe) document the post-G3-H finding that Internal Error **does not actually fire in operation** on the project's hardware once UncMask is cleared — the `Br_AER_Cor=0x1` and `GPU_Cor=0x2000` that motivated the patch were **stale RW1C bits from PCI enumeration**, not live error firings. The implication: C2 is **preventive observability hardening**, not a fix for actively-firing-but-silently-demoted errors. This is a non-obvious narrative inversion that bears on how an upstream reviewer might read the patch ("if Internal Errors don't fire, why unmask them?"). The intent's Purpose currently frames the value as "make subsequent Internal Errors VISIBLE" — which is technically accurate (forward-looking visibility) but doesn't surface the "we don't see them post-clear" empirical observation that completes the upstream story. Worth considering as an intent refinement but the v2 review's Rationale (review lines 14-29) and the design-choices section already establish the mechanism without claiming actively-firing errors; the framing gap is small.
- **Constraint discovered — surgical narrowing came AFTER the empirical journey, not before.** `patches/0022-G3-H-clear-AER-UncMask-match-Windows.patch:60-90` cleared the entire `PCI_ERR_UNCOR_MASK` to zero (Windows-parity rationale, no narrowing). Legacy P5 (`/root/nvidia-driver-injector/patches/legacy/0002-tb-egpu-aer-uncmask-clear.patch:236-251`) likewise cleared the whole mask. The narrowing to `PCI_ERR_UNC_INTN` + `PCI_ERR_COR_INTERNAL` only appeared in the v1 C2 commit (`2cc240ef`, 2026-05-22) after the May 2026 source review surfaced kernel 7.0's `pci_aer_unmask_internal_errors()` as the canonical operation. v2 review records this as the strongest single architectural improvement against the legacy ancestors (review lines 113-121). v3 review concurs: narrowing is correct, kernel-canonical, and preserves any platform-deliberate masks on other bits. **No refinement candidate** — the narrowing decision is durably captured.
- **Alternatives considered + rejected — gated module parameter.** Legacy P5 had `NVreg_TbEgpuAerUncMaskClear` (default 1). G3-H had `NVreg_TbEgpuUncMaskClearEnable`. v1 has neither. The aorus author's reasoning at `/root/nvidia-driver-injector/patches/legacy/0002-tb-egpu-aer-uncmask-clear.patch:175-184`: "Default-on. Disable with `NVreg_TbEgpuAerUncMaskClear=0` if it causes regression on a specific board." The v2 review's Design choices section (review lines 196-203) explicitly removes the parameter for upstream-friendliness: "the operation is surgical enough (two bits, on the probed PF, idempotent) that an unconditional default is correct. If a regression ever surfaces, a knob can be added in the addon layer [[A5-version-and-toggles]] without re-touching this patch." This carve mirrors the C1/A5 carve pattern (C-set = upstream-pure; A-set = project-local toggles). **No refinement candidate** — the unconditional-with-A5-escape-hatch posture is durably captured in both the intent's Scope boundary (intent lines 104-109) and the v2 review.
- **Alternatives considered + rejected — bound-checking the AER ext-cap offset.** Legacy P5 (`patches/legacy/0002-tb-egpu-aer-uncmask-clear.patch:195-220`) bound-checked `aer_pos` against `[0x100, 0x1000)` and emitted `dev_warn` on out-of-range with a `-ERANGE` short-circuit. Reasoning (the in-file comment lines 187-194): "PCIe spec puts extended capabilities at offsets >= 0x100 and < 0x1000. `pci_find_ext_capability()` returns a value within that range on success and 0 if absent; we treat any value outside [0x100, 0x1000) as broken hardware reporting and skip the write to avoid corrupting unrelated config-space." v1 C2 dropped this — `pci_find_ext_capability` enforces the spec internally; an out-of-range return would be a kernel bug, not nvidia.ko's concern. The kernel's own `pci_aer_unmask_internal_errors()` (linux-v6.19 `drivers/pci/pcie/aer.c:1135-1147`) does not bound-check either. **No refinement candidate** — the bound-check would add defense against an impossibility relative to the kernel's contract, and would diverge from the kernel-canonical operation's shape; default-reject for upstream-bound surface.
- **Alternatives considered + rejected — `dev_warn` on config-space read/write failure.** Legacy P5 (`patches/legacy/0002-tb-egpu-aer-uncmask-clear.patch:222-244`) warned on `pci_read_config_dword` and `pci_write_config_dword` failure. v1 C2 is silent on both. v2 review's Design choices section (review lines 205-216) records the deliberate choice: "(1) `pci_read_config_dword` on a present device is extremely unlikely to fail at probe time — when it does, the device is already in such bad shape that other probe steps will fail visibly almost immediately; (2) adding warn paths grows the upstream diff for low-value signal." The kernel function also silently ignores read/write returns. **No refinement candidate** — the silence is intentional and matches kernel idiom.
- **Forgotten / latent invariant — `pcie_aer_is_native()` precondition.** The kernel's `pci_aer_unmask_internal_errors()` kerneldoc (linux-v6.19 `drivers/pci/pcie/aer.c:1131-1134`) explicitly states "AER must be enabled and supported by the device which must be checked in advance, e.g. with `pcie_aer_is_native()`." On systems where firmware (rather than the OS) owns AER (i.e. no `_OSC` AER handoff), writing the mask is potentially ineffective — firmware may rewrite the mask on the next AER event. v1 C2 does not check this; the intent does not surface it as a Scope boundary item. The companion symbol `pcie_aer_is_native` is `EXPORT_SYMBOL_NS_GPL(pcie_aer_is_native, "CXL")` (`drivers/pci/pcie/aer.c:254`) — namespace-restricted to CXL, so even if nvidia.ko wanted to check the precondition, the symbol is unreachable. The legacy P5 does not check this either. **Forgotten invariant** — but operationally fine: the mask write is harmless on firmware-AER systems (worst case the unmask is a no-op there), and the intent's "MUST NOT change either mask register if the AER extended capability is absent" (intent line 41-42) covers the absence case. Surfacing this as an intent clause adds complexity for a hardware regime C2 can't reach anyway. See I3 below for the triage.
- **Forgotten / latent invariant — read failure path safety.** v1 C2 checks `pci_read_config_dword(...) == 0 && (mask & PCI_ERR_UNC_INTN) != 0`. If the read fails (rc != 0), `mask` is uninitialized — but the short-circuit `&&` skips the bit-test before reading `mask`, so the uninitialized value is never inspected. This is sound C. The kernel's own function does NOT check read return at all (linux-v6.19 `drivers/pci/pcie/aer.c:1140`); on read failure `mask` would be whatever the read primitive left in it (typically 0xFFFFFFFF on bus failure), and the subsequent `mask &= ~PCI_ERR_UNC_INTN; pci_write_config_dword(...)` would write 0xFFBFFFFF — corrupting the mask register with all-ones-except-the-cleared-bit. v1 C2's defensive read-return check is **strictly safer than the kernel idiom** under a bus-disconnect-during-probe scenario, even though that scenario is improbable at probe time. This safety property is implicit in the code and absent from the intent. Could be lifted but very low-stakes — see I4 below.
- **Forgotten / latent invariant — VF exclusion is enforced by the SR-IOV early-exit, not by the helper.** Intent Requirement 2 specifies the VF-exclusion contract; v1 enforces it by placing the call site after the existing `NV_PCI_SRIOV_SUPPORT` block. The helper itself does not check `pci_dev->is_virtfn`. This is captured in the v1 call-site comment (`nv-pci.c:1855-1858` of the patched file) and in v2 review's Strengths section (review lines 128-131). **No refinement candidate** — defense-in-depth (also checking `is_virtfn` inside the helper) would be over-defensive against a contract the existing surrounding code already enforces.

## Improvements considered

### C2-aer-internal-unmask-I1 — Add bound-check on the AER ext-cap offset

- **Lens:** robustness (re-considered against the legacy P5 bound-check)
- **Current state:** v1 helper uses `pci_find_ext_capability`'s return directly:
  ```c
  int aer = pci_find_ext_capability(pci_dev, PCI_EXT_CAP_ID_ERR);
  ...
  pci_read_config_dword(pci_dev, aer + PCI_ERR_UNCOR_MASK, &mask)
  ```
  No bound-check on `aer`. The legacy P5 (`patches/legacy/0002-tb-egpu-aer-uncmask-clear.patch:195-220`) bound-checked `aer_pos` against `[0x100, 0x1000)` with `dev_warn` + `-ERANGE` short-circuit, calling broken-hardware-reporting out-of-range values.
- **Proposed state:** Reintroduce the bound-check from the legacy ancestor.
- **Value:** Defense against a hypothetical broken-hardware report of an out-of-range AER offset.
- **Cost:** +~6 LoC of paranoia. The kernel's `pci_find_ext_capability` enforces the `[0x100, 0x1000)` PCIe spec range internally — an out-of-range return would be a kernel bug. The kernel's own `pci_aer_unmask_internal_errors()` (linux-v6.19 `drivers/pci/pcie/aer.c:1135-1147`) does NOT bound-check either; uses `dev->aer_cap` directly. Adding the check diverges from the kernel-canonical shape that the upstream PR posture leans on.
- **Verification mode:** A (code-reading; would need to read kernel `pci_find_ext_capability` to confirm internal enforcement).
- **Intent impact:** none (defensive code; intent doesn't constrain offset-validation behaviour).
- **Triage decision:** reject.
- **Resolution:** rejected — the bound-check defends against an impossibility relative to the kernel's `pci_find_ext_capability` contract; diverges from the kernel's own `pci_aer_unmask_internal_errors()` shape that C2 is consciously paralleling for upstream-PR posture; legacy P5's paranoia was project-internal preference not empirical necessity. The aorus author's in-file comment at P5 lines 187-194 frames it as defensive-against-broken-hardware-reporting; v3 review concurs that this is over-defensive for an upstream-bound patch.

### C2-aer-internal-unmask-I3 — Surface `pcie_aer_is_native()` precondition as a Scope boundary clause

- **Lens:** invariant clarity (forgotten kernel-contract precondition)
- **Current state:** The kernel's `pci_aer_unmask_internal_errors()` kerneldoc (linux-v6.19 `drivers/pci/pcie/aer.c:1131-1134`) explicitly names the precondition: "AER must be enabled and supported by the device which must be checked in advance, e.g. with `pcie_aer_is_native()`." - **Proposed state:** Add one Scope-boundary clause noting that C2's unmask is a config-space write whose effectiveness depends on the OS holding AER ownership (via `_OSC` handoff); on firmware-AER systems the write is harmless but may be no-op'd by firmware on the next AER event. C2 deliberately does NOT call `pcie_aer_is_native()` (kernel symbol is namespace-restricted to CXL).
- **Value:** Surfaces a forgotten kernel-contract precondition that becomes load-bearing on hardware C2 wasn't tested against (firmware-AER systems aren't this project's hardware). Helps an upstream reviewer see C2's scope is correctly limited.
- **Cost:** ~3 lines added to intent's Scope boundary. Re-opens intent's `reviewed` lint state. The hardware regime (firmware-AER) is one C2 can't meaningfully address anyway — surfacing the limitation without acting on it adds intent surface for marginal value.
- **Verification mode:** A.
- **Intent impact:** add Scope boundary clause.
- **Triage decision:** reject.
- **Resolution:** rejected — the mask write is *harmless* on firmware-AER systems (worst case the unmask is a no-op there; nothing is corrupted), so the limitation is a non-event for the patch's correctness contract. The kernel function's `pcie_aer_is_native()` precondition is a CXL-subsystem invariant (CXL needs to know it's allowed to handle the error); the analogous nvidia.ko question ("did our unmask stick?") is empirically observable via `pci_error_handlers` being invoked or not — C4's `error_detected` registration is the natural detection surface, not a probe-time precondition check. The legacy P5 didn't surface this either; neither did the aorus author's design notes. Default-reject: low-value precision over a hardware regime the patch can't reach, and unmentioned in the proven ancestor.

### C2-aer-internal-unmask-I4 — Lift the "read-failure safety via short-circuit `&&`" invariant from review prose into intent

- **Lens:** invariant clarity (subtle correctness property)
- **Current state:** v1 helper checks `pci_read_config_dword(...) == 0 && (mask & PCI_ERR_UNC_INTN) != 0`. On read failure, `mask` is uninitialized but the short-circuit `&&` skips the bit-test, so the uninitialized value is never inspected. This is sound C. The kernel's own function does NOT check read return at all (linux-v6.19 `drivers/pci/pcie/aer.c:1140`), making v1 C2 **strictly safer than kernel idiom** under a bus-disconnect-during-probe scenario. The intent does not surface this; the v2 review captures it as a Strength (review lines 132-135 "Reads-before-writes ... no unnecessary config-space writes, no log noise") but not as an explicit safety claim.
- **Proposed state:** Add to intent's Scope boundary or as a Requirement note: "On `pci_read_config_dword` failure, the helper SHALL NOT write `PCI_ERR_UNCOR_MASK` or `PCI_ERR_COR_MASK`. This is enforced by the short-circuit `&&` in the bit-test expression."
- **Value:** Captures the subtle short-circuit-based safety property explicitly, before a future maintainer "simplifies" the expression and introduces the bug.
- **Cost:** ~2 lines added to intent. Re-opens intent's `reviewed` lint state. The property is also enforceable by code review and is obvious from reading the conditional (a future maintainer who removes the read-return check would be obviously dropping a guard).
- **Verification mode:** A.
- **Intent impact:** refine Scope boundary or add Requirement note.
- **Triage decision:** defer.
- **Resolution:** deferred — the short-circuit safety is locally evident from the code (the `&&` is right there in the expression) and the v2 review's Strengths section (review lines 132-135) names "Reads-before-writes" as a deliberate design property. Lifting an obvious-from-the-code property into the intent inflates the intent for a guard a code-reviewer would catch. **Disposition for follow-up:** if a future patch refactors the helper (e.g. extracts the read into a separate statement), revisit and surface the invariant then. Tracked here so a future maintainer doesn't drop the guard.

### C2-aer-internal-unmask-I5 — Use `pci_dev->aer_cap` cached field instead of `pci_find_ext_capability`

- **Lens:** performance / quality (mirror kernel idiom)
- **Current state:** v1 helper calls `pci_find_ext_capability(pci_dev, PCI_EXT_CAP_ID_ERR)` which walks the PCI extended capability list (~30+ config reads worst-case). The kernel's own `pci_aer_unmask_internal_errors()` (linux-v6.19 `drivers/pci/pcie/aer.c:1137`) uses `int aer = dev->aer_cap;` — the cached field populated by `pci_aer_init()` (linux-v6.19 `drivers/pci/pcie/aer.c:392`) which runs in `pci_setup_device` BEFORE driver probe is invoked (linux-v6.19 `drivers/pci/probe.c:2701`).
- **Proposed state:** Replace the `pci_find_ext_capability` call with `int aer = pci_dev->aer_cap;` to match the kernel idiom and skip the cap-list walk.
- **Value:** ~30 fewer PCI config reads per probe (one-shot, probe-only — micro-optimization). Matches the kernel-canonical idiom that the patch is consciously paralleling.
- **Cost:** `pci_dev->aer_cap` is `u16` (zero when AER absent — same semantics as `pci_find_ext_capability == 0` check).
